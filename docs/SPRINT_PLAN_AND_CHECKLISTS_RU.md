# Спринты, задачи и чек-листы по проекту

Дата: 19 мая 2026

## 1. Назначение документа

Документ фиксирует:

- разбивку проекта по спринтам;
- цели каждого спринта;
- состав задач;
- чек-листы готовности;
- зависимости между этапами;
- перечень доступов и артефактов, которые нужно запросить перед разработкой.

Этот документ является рабочим roadmap для старта реализации.

## 2. Общая цель проекта

Собрать серверную систему, которая:

- слушает новые заявки во внешней платформе;
- быстро пытается их захватывать;
- умеет держать несколько активных заявок одновременно;
- управляется через Telegram-бота;
- работает на `Ubuntu` через `Docker`;
- сохраняет историю заявок в БД;
- использует `Playwright` только как fallback и для reauth/captcha-сценариев.

## 3. Рабочая модель проекта

Рекомендуемая длина одного спринта:

- `5-7` рабочих дней.

Рекомендуемая структура delivery:

- `Sprint 0` — подготовка и доступы;
- `Sprint 1` — исследование платформы и протокола;
- `Sprint 2` — backend skeleton и инфраструктура;
- `Sprint 3` — socket listener и claim engine;
- `Sprint 4` — Telegram control-plane и multi-order flow;
- `Sprint 5` — Playwright fallback, reauth и captcha flow;
- `Sprint 6` — стабилизация, тесты, деплой и запуск.

## 4. Что нужно запросить заранее

До активной разработки желательно получить:

- доступ к платформе;
- тестовый аккаунт;
- рабочий аккаунт, если тестового нет;
- описание ролей пользователя;
- возможность безопасно логиниться с сервера или с отдельной машины;
- пример реальной новой заявки;
- доступ к веб-интерфейсу, где появляются заявки;
- понимание, можно ли держать несколько активных заявок одновременно;
- подтверждение лимита, например `3` активные заявки;
- описание, что считается завершением заявки;
- список обязательных ручных действий оператора;
- понимание, где и когда появляется капча;
- предварительное понимание, решаем ли капчу вручную, через локальный эксперимент или внешний сервис;
- понимание, истекает ли `access token`, и как часто.

## 10. Sprint 4 — Telegram Bot и multi-order управление

### Цель

Собрать операторский контур управления агентом.

### Задачи

- реализовать `/run`;
- реализовать `/pause`;
- реализовать `/resume`;
- реализовать `/status`;
- реализовать `/relogin`;
- реализовать callback `Оплачено`;
- реализовать двухшаговое подтверждение оплаты;
- реализовать уведомления о захвате;
- реализовать уведомления о заполнении лимита активных заявок;
- реализовать уведомления об освобождении слота;
- реализовать отображение списка активных заявок;
- реализовать guard по `telegram_user_id`.

### Бизнес-задачи

- при `Оплачено` заявка уходит из active set;

- поиск не должен останавливаться, пока не достигнут лимит.

### Результат спринта

- оператор полноценно управляет агентом через Telegram;
- `multi-order` сценарий работает через интерфейс бота.

### Задачи

- реализовать browser bootstrap;
- реализовать хранение browser session state;
- реализовать browser-assisted login;
- реализовать ручной reauth flow;
- реализовать обнаружение капчи;
- реализовать интерфейс `CaptchaSolverProvider`;
- реализовать primary captcha-solving provider;
- реализовать backup captcha-solving provider, если используется второй сервис;

- реализовать перевод агента в `captcha_required`;
- реализовать уведомление оператора о капче;

- реализовать возврат в рабочий режим после ручного прохождения капчи.

### Важный результат

Этот спринт не должен ломать direct request path.

Playwright нужен как:

- fallback;
- канал reauth;
- инструмент ручной стабилизации.

- `Sprint 2` блокирует production-работу по `Sprint 3-6`.

## 16. Что считать MVP

MVP достигнут, если:

- агент запускается на `Ubuntu` через `Docker Compose`;
- умеет слушать новые заявки;
- умеет брать заявки прямым запросом;
- умеет держать несколько активных заявок одновременно;
- умеет принимать `Оплачено` через Telegram;
- после `Оплачено` освобождает слот;
- не удаляет завершенные заявки из БД;
- умеет переживать разрыв сокета;
- умеет входить в режим ручного вмешательства при капче.

## 17. Финальный practical roadmap

Порядок работ, который я рекомендую:

1. Сначала закрыть доступы и реальные ограничения платформы.
2. Потом зафиксировать сокет и request на захват.
3. Затем довести инфраструктурный каркас.
4. После этого делать direct claim path и active set.
5. Затем собрать Telegram control-plane.
6. Только потом доводить Playwright fallback и captcha flow.
7. В конце стабилизировать и деплоить на сервер.
