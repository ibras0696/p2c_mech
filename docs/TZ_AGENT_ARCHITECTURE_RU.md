# ТЗ и архитектура системы автоматического захвата заявок

Дата: 16 мая 2026

## 1. Назначение документа

Документ описывает целевую архитектуру, алгоритм работы, технические требования и MVP-план для системы автоматического обнаружения и захвата новых заявок с управлением через Telegram-бота.

Документ рассчитан на серверное развертывание в `Ubuntu` с использованием `Docker`.

## 2. Цель проекта

Разработать серверного агента, который:

- непрерывно ожидает появления новых заявок;
- максимально быстро реагирует на новую подходящую заявку;
- автоматически инициирует захват заявки;
- после захвата продолжает работать как сервис и не завершает процесс;
- может одновременно вести несколько активных заявок;
- не останавливает поиск новых заявок, пока не достигнут лимит активных задач;
- после ручного подтверждения оператора `Оплачено` снимает конкретную заявку из активного пула, но сохраняет ее в БД и истории;
- управляется через Telegram-бота на `aiogram 3+`;
- устойчив к переподключениям, истечению токенов и ошибкам сети;
- поддерживает режимы обработки капчи `manual_only` и `auto_then_human`.

## 3. Бизнес-контекст

Ключевой KPI системы:

- минимальная задержка между появлением новой заявки и попыткой ее захвата.

Особенности процесса:

- заявка появляется во внешней платформе;
- агент должен как можно быстрее ее обнаружить;
- после захвата оператор вручную обрабатывает заявку;
- агент продолжает искать следующие заявки параллельно с уже взятыми;
- одновременно в работе может быть несколько заявок, например `3`;
- после действия `Оплачено` заявка убирается из активного набора, но остается в БД и истории.

## 4. Основные ограничения

### 4.1 Технические ограничения

- внешний сервис использует `access token`;
- источник событий, вероятно, построен на `WebSocket`;
- UI может содержать капчу;
- captcha-solving сервисы используются только в разрешенном режиме и только как опциональный контур;
- точный приватный API платформы заранее не гарантирован и должен быть исследован.

### 4.2 Архитектурные ограничения

- нельзя строить систему только на DOM-кликах, если критична скорость;
- нельзя полагаться только на браузер как единственный источник событий;
- нельзя считать webhook или UI обновление самым быстрым каналом, если есть прямой сокет;
- авторизация в Telegram и авторизация в целевой платформе должны быть разделены.

### 4.3 Операционные ограничения

- система будет работать на сервере под `Ubuntu`;
- первичная разработка и локальные тесты могут выполняться на `Windows`;
- финальная проверка runtime должна выполняться на `Ubuntu`;
- запуск должен быть контейнеризирован;
- управление должно выполняться удаленно, без постоянного ручного доступа к серверу;
- при появлении капчи должен включаться режим ручного вмешательства.

## 5. Ключевая архитектурная идея

Базовая архитектура должна строиться по гибридной модели:

- `WebSocket` используется как самый быстрый канал обнаружения новых заявок;
- прямой `HTTP/API request` используется как основной механизм захвата заявки;
- `Playwright` используется как fallback-исполнитель и инструмент восстановления сессии;
- `Telegram Bot` используется как управляющий интерфейс для оператора;
- вся логика живет в постоянном backend-сервисе.

Это предпочтительнее, чем схема `только Playwright`, по следующим причинам:

- сокет дает более раннее событие, чем визуальное появление в интерфейсе;
- backend-запрос быстрее клика по UI;
- Playwright полезен как резерв и для сложных сценариев авторизации;
- при частичной нестабильности приватного API можно не терять всю систему.

## 6. Целевая функциональность

Система должна уметь:

- запускаться как постоянный сервис;
- авторизовываться во внешней платформе;
- получать и хранить `access token`;
- при необходимости обновлять сессию;
- подключаться к `WebSocket` и слушать поток новых заявок;
- фильтровать заявки по заданным правилам;
- мгновенно пытаться захватить подходящую заявку;
- держать несколько активных заявок одновременно до заданного лимита;
- продолжать поиск новых заявок, пока есть свободные слоты;
- принимать команду `Оплачено` от оператора через Telegram;
- после `Оплачено` удалять заявку только из активного состояния агента, без удаления из БД;
- не предоставлять штатную кнопку `Пропустить` для уже захваченной заявки;
- останавливаться, ставиться на паузу и возобновляться;
- уведомлять оператора о капче, ошибках, потере токена, переподключении сокета и смене состояния.

## 7. Основные сценарии работы

### 7.1 Основной сценарий

1. Сервис запускается.
2. Агент проходит авторизацию или восстанавливает существующую сессию.
3. Агент подключается к `WebSocket`.
4. Агент переходит в состояние `waiting`.
5. При получении события о новой заявке выполняется фильтрация.
6. Если заявка подходит, агент переходит в `claiming`.
7. Выполняется попытка захвата:
   - сначала через прямой backend request;
   - при необходимости через `Playwright fallback`.
8. Если заявка захвачена, она попадает в список активных заявок агента.
9. Оператор получает уведомление в Telegram.
10. Если число активных заявок меньше лимита, агент продолжает искать следующие заявки без остановки.
11. После ручного действия `Оплачено` агент очищает только активный контекст конкретной заявки.
12. Заявка сохраняется в БД как завершенная или оплаченная.
13. Если после этого остается свободный слот, агент продолжает поиск в обычном режиме.

### 7.2 Если заявка не подошла

1. Событие новой заявки получено.
2. Правила фильтрации возвращают `reject`.
3. Агент игнорирует заявку.
4. Состояние остается `waiting`.

### 7.3 Если захват не удался

1. Событие новой заявки получено.
2. Заявка прошла фильтр.
3. Попытка `claim` завершилась ошибкой или отказом.
4. Агент пишет результат в журнал.
5. Агент остается в рабочем режиме и продолжает прием новых событий, если есть свободные слоты.

### 7.4 Если появилась капча

1. Во время авторизации, обновления сессии или browser fallback обнаруживается капча.
2. Агент переводится в `captcha_required`.
3. Захват новых заявок блокируется.
4. Оператор получает уведомление в Telegram.
5. После ручного решения капчи агент восстанавливает работу.

## 8. Архитектурные принципы

### 8.1 Event-driven

Система должна быть событийной:

- входные события приходят из `WebSocket`;
- внутренние команды приходят из Telegram-бота;
- переходы состояния выполняются через внутреннюю event bus-модель.

### 8.2 State machine

Поведение агента должно быть описано через явную машину состояний, а не через набор флагов.

При этом нужно разделить:

- общее operational state агента;
- lifecycle отдельных заявок.

### 8.3 Fail-safe

При неясной ситуации агент должен:

- не брать новую заявку;
- не пытаться повторно брать уже захваченную заявку без проверки;
- переводиться в безопасное состояние и уведомлять оператора.

### 8.4 Human-in-the-loop

Система не должна пытаться обходить капчу автоматически.

При наличии капчи:

- агент приостанавливает критический путь;
- оператор вмешивается вручную;
- после подтверждения работа продолжается.

### 8.5 Separation of concerns

Нужно развести:

- Telegram-управление;
- бизнес-логику агента;
- авторизацию;
- сокет;
- механизм захвата;
- браузерный fallback;
- хранилище состояния.

## 9. Компоненты системы

### 9.1 Telegram Bot

Технология:

- `Python`
- `aiogram 3+`

Роль:

- управление агентом;
- отображение состояния;
- уведомления;
- ручные команды оператора.

Функции:

- `/start`
- `/run`
- `/pause`
- `/resume`
- `/status`
- `/relogin`
- просмотр текущего лимита активных заявок
- изменение лимита активных заявок
- кнопка `Проверить оплату`
- экран подтверждения `Подтвердить оплачено`
- кнопка `Переавторизация выполнена`

### 9.2 Control Service

Роль:

- главный orchestration-компонент;
- принимает сигналы от Telegram-бота;
- принимает события от сокета;
- вызывает Claim Engine;
- хранит текущее состояние агента.

### 9.3 Auth Manager

Роль:

- получение `access token`;
- хранение сессии;
- восстановление сессии;
- обновление токена;
- синхронизация браузерной и API-сессии, если это необходимо.

Функции:

- проверка валидности токена;
- refresh или повторный login;
- выдача актуального auth context остальным компонентам.

### 9.4 Socket Listener

Роль:

- постоянное подключение к `WebSocket`;
- heartbeat;
- автоматическое переподключение;
- получение событий новых заявок.

Функции:

- auth handshake;
- подписка на нужный канал;
- нормализация payload;
- передача события в очередь обработки.

### 9.5 Claim Engine

Роль:

- фильтрация заявок;
- защита от дублей;
- принятие решения о захвате;
- выполнение захвата;
- контроль лимита параллельных активных заявок;
- управление переходом `waiting -> claiming` и наполнением пула активных заявок.

### 9.6 API Client

Роль:

- прямое выполнение backend request на захват заявки;
- чтение ошибок платформы;
- работа с таймаутами;
- повторная попытка только по разрешенным сценариям.

### 9.7 Playwright Worker

Роль:

- browser fallback;
- ручная или полуавтоматическая авторизация;
- восстановление сессии;
- действия в UI, если API-сценарий недоступен.

Важно:

- `Playwright` не должен быть единственным критическим механизмом захвата;
- он должен использоваться как fallback или auxiliary channel.

Дополнительно:

- `CloakBrowser` может быть исследован как опциональный browser runtime для fallback-контура;
- `CloakBrowser` не считается решателем капчи;
- включать его в production default можно только после сравнения с обычным Playwright Chromium на Windows и Ubuntu.

### 9.8 Storage

Роль:

- хранение конфигурации;
- хранение истории событий;
- хранение статуса агента;
- хранение токенов и следов авторизации;
- хранение dedup-идентификаторов.

Рекомендуемое разделение:

- `PostgreSQL` для постоянных данных;
- `Redis` для быстрых состояний, блокировок и ephemeral queues.

## 10. Состояния агента

Рекомендуемая state machine агента:

- `offline`
- `authenticating`
- `waiting`
- `claiming`
- `capacity_reached`
- `paused`
- `captcha_required`
- `reauth_required`
- `error`

Описание:

- `offline` — сервис не работает или не инициализирован.
- `authenticating` — идет логин, refresh или восстановление сессии.
- `waiting` — агент слушает новые заявки и готов к захвату, есть хотя бы один свободный слот.
- `claiming` — агент выполняет попытку захвата.
- `capacity_reached` — достигнут лимит активных заявок, новые заявки временно не захватываются.
- `paused` — захват новых заявок приостановлен вручную.
- `captcha_required` — требуется ручное прохождение капчи.
- `reauth_required` — токен недействителен и нужна переавторизация.
- `error` — критическая ошибка, требующая диагностики.

### 10.1 Допустимые переходы

- `offline -> authenticating`
- `authenticating -> waiting`
- `authenticating -> captcha_required`
- `authenticating -> reauth_required`
- `waiting -> claiming`
- `claiming -> waiting`
- `claiming -> capacity_reached`
- `waiting -> capacity_reached`
- `capacity_reached -> waiting`
- `waiting -> paused`
- `paused -> waiting`
- `capacity_reached -> paused`
- `captcha_required -> authenticating`
- `reauth_required -> authenticating`
- `any -> error`

### 10.2 Состояния отдельной заявки

Каждая заявка должна иметь собственный lifecycle:

- `detected`
- `claiming`
- `claimed`
- `owned_confirmed`
- `in_progress`
- `paid`
- `completed`
- `claim_uncertain`
- `failed`

Практическое правило:

- агент может иметь несколько заявок в статусе `in_progress` одновременно;
- нажатие `Оплачено` переводит конкретную заявку в `paid` или `completed`;
- после этого заявка удаляется только из оперативного active set, но не удаляется из БД.
- карточка оплаты в Telegram создается только после `owned_confirmed`.

## 11. Паттерны проектирования

Ниже перечислены паттерны, которые должны использоваться в проекте.

### 11.1 State Pattern

Нужен для явного описания поведения агента в зависимости от состояния.

Применение:

- переходы между `waiting`, `claiming`, `capacity_reached`, `paused`, `captcha_required`.

Польза:

- меньше скрытых флагов;
- проще отлаживать;
- проще ограничивать недопустимые команды.

### 11.2 Strategy Pattern

Нужен для выбора способа захвата заявки.

Стратегии:

- `DirectApiClaimStrategy`
- `PlaywrightClaimStrategy`

Польза:

- можно сначала пытаться захватить через API;
- при необходимости переключаться на browser fallback;
- легко расширять поведение без переписывания всего engine.

### 11.3 Circuit Breaker

Нужен для защиты от лавины неудачных запросов.

Применение:

- API платформы отвечает ошибками;
- сокет часто рвется;
- авторизация многократно не проходит.

Польза:

- система не начнет бомбить внешний сервис ошибочными запросами;
- оператор быстрее увидит, что агент находится в деградировавшем режиме.

### 11.4 Retry with Backoff

Нужен для безопасных повторов.

Применение:

- reconnect сокета;
- временные сетевые ошибки;
- non-fatal ошибки чтения состояния.

Важно:

- повтор нельзя делать бездумно на `claim`, если может возникнуть двойной захват;
- повтор на критическом действии допустим только при наличии идемпотентности или ясного статуса ответа.

### 11.5 Repository Pattern

Нужен для отделения бизнес-логики от SQL/Redis-слоя.

Применение:

- `AgentStateRepository`
- `OrderRepository`
- `TokenRepository`
- `EventLogRepository`

### 11.6 Observer / Pub-Sub

Нужен для событийной связи компонентов.

События:

- `order_detected`
- `claim_started`
- `claim_succeeded`
- `claim_failed`
- `agent_paused`
- `captcha_detected`
- `token_expired`

### 11.7 Command Pattern

Нужен для обработки команд из Telegram.

Команды:

- `RunAgentCommand`
- `PauseAgentCommand`
- `MarkPaidCommand`
- `ResumeAfterCaptchaCommand`

## 12. Логика авторизации

### 12.1 Разделение авторизации

Есть 2 разных уровня авторизации:

- авторизация оператора в Telegram-боте;
- авторизация агента во внешней платформе.

Они не должны смешиваться.

### 12.2 Авторизация оператора

Бот должен принимать команды только от разрешенных `telegram_user_id`.

Нужно предусмотреть:

- список администраторов;
- список операторов;
- журнал команд.

### 12.3 Авторизация агента в платформе

Нужно поддержать:

- логин с получением `access token`;
- хранение токена и связанной сессии;
- refresh, если он поддерживается;
- повторную авторизацию при необходимости.

### 12.4 Требования к токенам

- токены нельзя хранить в открытом виде в логах;
- токены должны храниться только в защищенном storage;
- доступ к ним должен быть ограничен сервисным контуром;
- при истечении токена агент не должен продолжать "слепой" захват заявок.

## 13. Логика работы с капчей

### 13.1 Базовое правило

Базовый штатный режим — ручное прохождение капчи оператором.

Автоматическое решение капчи допускается только как отдельный опциональный контур, если это не нарушает правила платформы и принято как осознанный риск проекта.

### 13.2 Если капча возникает только на логине

Подход:

- использовать долгоживущую сессию;
- ручной login выполнять редко;
- далее переиспользовать токен и cookies.

### 13.3 Если капча иногда появляется при подозрительной активности

Подход:

- агент переводится в `captcha_required`;
- Telegram-бот уведомляет оператора;
- оператор вручную проходит капчу в браузерной сессии;
- агент возвращается в `waiting`.

### 13.4 Если капча регулярно появляется при захвате заявки

Риск:

- если капча часто появляется именно на `claim`, система становится зависимой от качества captcha-solving provider или ручного оператора.

Следствие:

- это нужно зафиксировать как риск проекта;
- такой сценарий может сделать KPI по скорости недостижимым.

### 13.5 Опциональный контур Captcha Solver

Система должна предусматривать абстрактный слой `CaptchaSolverProvider`.

Цель:

- попробовать автоматически решить капчу через 1-2 внешних провайдера;
- не привязывать бизнес-логику к конкретному сервису;
- при неуспехе автоматически переводить сценарий на ручного оператора.

Рекомендуемая схема:

- `PrimaryCaptchaSolverProvider`
- `BackupCaptchaSolverProvider`
- `HumanCaptchaFallbackProvider`

Провайдеры-кандидаты для исследования:

- `2Captcha`
- `CapSolver`
- `CapMonster Cloud`

Ограничения:

- ключи провайдеров хранятся только в `.env` или secret storage;
- автоматическое решение включается только через конфиг;
- все попытки решения логируются;
- если провайдер вернул ошибку, превышен таймаут или капча не принята платформой, агент переходит в `captcha_required`;
- ручной оператор остается обязательным fallback.

Рекомендуемые режимы:

- `manual_only` — только ручное прохождение;
- `auto_then_human` — сначала внешний solver, затем оператор;
- `disabled` — контур капчи полностью выключен.

Для MVP разумный режим:

- `auto_then_human`, если принято решение использовать внешний captcha provider;
- `manual_only`, если нет подтверждения, что это допустимо для платформы.

## 14. Алгоритм обработки новой заявки

### 14.1 Быстрый путь

1. Сокет присылает событие новой заявки.
2. Событие нормализуется.
3. Проверяется dedup по `order_id`.
4. Проверяется текущее состояние агента и число активных заявок.
5. Если агент в `paused`, `captcha_required`, `reauth_required` или `error`, событие логируется и не обрабатывается.
6. Если достигнут лимит активных заявок, агент фиксирует состояние `capacity_reached` и не делает новый `claim`.
7. Если есть свободный слот, заявка прогоняется через фильтры.
8. Если заявка подходит, агент атомарно резервирует слот и переходит в `claiming`.
9. Выполняется `DirectApiClaimStrategy`.
10. После первичного успеха claim заявка не отправляется в Telegram, пока не подтвержден ownership.
11. Выполняется вторая проверка по `order_id`: заявка должна явно принадлежать нашему аккаунту (`owned_confirmed`).
12. Только после `owned_confirmed` заявка добавляется в active set, и оператор получает карточку с `QR` и `payment_link`.
13. Если ownership не подтвержден, заявка переходит в `claim_uncertain`/`failed`, слот освобождается, в Telegram заявка не отправляется.
14. Если после добавления достигнут лимит, агент переходит в `capacity_reached`, иначе возвращается в `waiting`.
15. Если API-захват невозможен по технической причине, запускается `PlaywrightClaimStrategy`.
16. Если обе стратегии неуспешны, слот освобождается, агент возвращается в `waiting` или остается в `capacity_reached` по фактической загрузке.

### 14.2 Завершение заявки через `Оплачено`

1. Оператор открывает конкретную активную заявку.
2. Система показывает экран проверки оплаты с деталями заявки.
3. Оператор проверяет реквизиты, сумму, направление и факт оплаты.
4. Оператор нажимает `Подтвердить оплачено`.
5. Система находит заявку в active set.
6. Статус заявки переводится в `paid` или `completed`.
7. Заявка удаляется из оперативного списка активных заявок.
8. Запись о заявке, ее статусах и истории остается в БД.
9. Если освобожден слот, агент переходит в `waiting` и продолжает захват новых заявок.

Критическое правило:

- если заявка уже захвачена, оператор не должен иметь простую кнопку `Пропустить`;
- если заявка не оплачена, время блокировки считается операционным риском;
- чтобы не поймать чужую заявку, фильтрация до `claim` должна быть строгой, а ручная оплата выполняется только после проверки деталей заявки.
- при подтвержденном инциденте блокировки за долгую оплату нужно вести дедлайн оплаты по каждой активной заявке и оповещать оператора до истечения окна.

SLA по времени оплаты (по официальной базе знаний, 20 февраля 2026):

- если мерчант не оплатил и не отменил заказ в течение `3` минут, заказ автоотменяется;
- после такой автоотмены мерчант не может брать новые заказы `1` час.

Следствие для системы:

- после `owned_confirmed` запускать per-order countdown `180` секунд;
- Telegram-бот обязан показывать обратный отсчет и предупреждения (`T-60`, `T-30`, `T-10`);
- при риске просрочки новых заявок не брать, пока оператор не закроет текущую;
- оплачивать можно только заявки в статусе `owned_confirmed` с неистекшим дедлайном.

### 14.3 Антидубли

Должно быть:

- хранение уже увиденных `order_id`;
- защита от повторного захвата одной и той же заявки;
- mutex или distributed lock на переход в `claiming`.

## 15. Фильтрация заявок

Система должна поддерживать бизнес-правила фильтрации.

Примеры:

- сумма от и до;
- тип заявки;
- валюта;
- направление;
- география;
- приоритет;
- пользовательские признаки риска.

Требование:

- фильтрация должна выполняться до expensive-операций;
- фильтры должны быть конфигурируемыми.
- фильтрация должна быть строгой, потому что захват чужой заявки и последующая оплата приведет к уходу денег другому получателю.

## 32. Финальный вывод

Для данного проекта не рекомендуется строить систему как `Playwright-only`.

Оптимальная схема:

- `socket -> filter -> direct claim -> active pool -> paid/completed -> slot released`

с резервным browser-контуром:

- `Playwright` для fallback, login, reauth и работы с капчей.

Такая архитектура дает лучший баланс между:

- скоростью захвата;
- управляемостью;
- отказоустойчивостью;
- реалистичностью внедрения с ручным fallback при проблемах с капчей.
