# Sprint 0 — подготовка и входные условия

Дата старта: 19 мая 2026

## 1. Цель

Подготовить входные условия для разработки, чтобы Sprint 1 можно было начать с реальными доступами, понятным окружением и подтвержденными рабочими правилами.

## 2. Зафиксированные решения

### 2.1 Окружение разработки и тестирования

Решение:

- первичная разработка и локальные тесты выполняются на `Windows`;
- финальная проверка окружения, Docker и runtime выполняется на `Ubuntu`;
- production-цель остается `Ubuntu + Docker Compose`.

Практический вывод:

- локальный Windows-контур нужен для быстрой разработки;
- Ubuntu-контур нужен для проверки реального запуска сервиса.

### 2.2 Лимит активных заявок

Решение:

- агент должен поддерживать несколько активных заявок;
- стартовое значение можно принять как `3`, если не будет другого ограничения платформы;
- лимит должен управляться через Telegram-бота.

Что нужно реализовать позже:

- команда или кнопка просмотра текущего лимита;
- команда изменения лимита;
- сохранение лимита в БД;
- применение лимита без перезапуска сервиса.

### 2.3 Статусная модель

Под статусной моделью имеется в виду не отдельная сложная бюрократия, а понятные состояния:

- состояния агента;
- состояния заявки;
- правила переходов между ними.

Минимум для MVP:

- агент: `waiting`, `claiming`, `capacity_reached`, `paused`, `captcha_required`, `error`;
- заявка: `detected`, `claiming`, `claimed`, `in_progress`, `paid`, `completed`, `failed`.

Что нужно подтвердить после доступа:

- какие статусы есть во внешней платформе;
- какие статусы нам нужно хранить у себя;
- что именно происходит после `Оплачено`.

### 2.4 Роль Telegram-бота

Решение:

- бот остается обычным Telegram-ботом на `aiogram 3+`;
- его роль — пульт управления и уведомления;
- бизнес-логика живет в backend service, а не внутри handlers бота.

Минимальные функции бота:

- запуск агента;
- пауза;
- статус;
- просмотр активных заявок;
- изменение лимита активных заявок;
- кнопка `Оплачено` для конкретной заявки;
- уведомления об ошибках, капче, токене и сокете.

### 2.5 Docker Compose

Решение:

- `Docker Compose` обязателен;
- это основной способ запуска на Ubuntu;
- локально на Windows тоже можно использовать Compose, если окружение готово.

Сервисы:

- `app`;
- `bot`;
- `browser-worker`;
- `postgres`;
- `redis`.

### 2.6 `.env`

Решение:

- `.env` не хранится в репозитории;
- `.env.example` хранит только шаблон;
- локальный `.env` используется на Windows;
- отдельный production `.env` используется на Ubuntu;
- секреты не логируются.

Что нужно определить:

- кто хранит production `.env`;
- как передаются секреты на сервер;
- кто имеет доступ к Telegram token и платформенным учетным данным.

### 2.7 Капча

Решение на Sprint 0:

- автоматизацию капчи не фиксируем как обязательную;
- сначала выясняем реальный тип капчи и частоту появления;
- ручное прохождение оператором остается базовым fallback;
- `CloakBrowser` оставляем как опциональный кандидат для исследования, если он снизит частоту капчи.

Важно:

- `CloakBrowser` не считается решателем капчи;
- captcha-solving сервисы не добавляются в `.env` до отдельного решения.

## 6. Definition of Done для Sprint 0

Sprint 0 считается завершенным, если:

- доступы к платформе получены или явно назначен владелец их получения;
- Telegram bot token получен или назначен владелец его создания;
- список операторов и админов известен;
- подтвержден контур `Windows -> Ubuntu`;
- подтвержден обязательный `Docker Compose`;
- определен начальный лимит активных заявок;
- подтверждено, что лимит должен меняться через бота;
- описан ручной fallback по капче;
- подготовлены артефакты для reverse engineering в Sprint 1.
