# Разбор `@send / Crypto Bot` и его API

Дата анализа: 16 мая 2026  
Язык: русский  
Основано на официальной документации Crypto Bot / send.tg и Telegram-страницах бота.

## Короткий вывод

`@send` не является отдельным независимым продуктом с отдельным публичным API. Это пользовательский entrypoint и inline-алиас экосистемы `Crypto Bot`.

Для внешней интеграции официальный публичный API есть у подсистемы `Crypto Pay`. Через него можно:

- создавать инвойсы;
- принимать оплату;
- создавать чеки;
- делать выплаты пользователям;
- получать статусы и статистику;
- принимать webhook-события.

Для `P2P Market`, `Exchange` и полного управления кошельком пользователя официальной публичной API-документации для партнерской интеграции не найдено. В официальной базе знаний `P2P` описан как пользовательский интерфейс и пользовательский сценарий внутри бота, а не как открытый API для сторонних сервисов.

## Что такое `@send`

По официальной Telegram-странице `@send` запускает `Crypto Bot`. В описании указано, что через `@CryptoBot` можно покупать, продавать, хранить, отправлять и оплачивать криптовалютой внутри Telegram.

Практически это означает, что:

- `@send` используется как пользовательский alias;
- inline-команда `@send ...` применяется для создания чеков и инвойсов прямо из чата;
- бизнес-интеграции строятся не через какой-то отдельный "`@send API`", а через `Crypto Pay API`.

## Что входит в систему Crypto Bot

По официальному help center и сайту в экосистему входят:

- custodial wallet внутри Telegram;
- депозиты и выводы;
- checks;
- invoices;
- `Crypto Pay`;
- `P2P Market`;
- `Exchange`;
- `Subscriptions` для каналов и групп;
- web app: `https://app.send.tg`.

Важно: это централизованная custodial-система внутри инфраструктуры Crypto Bot. Это не self-custody API вашего проекта и не универсальный blockchain gateway.

## Ответ на вопрос про `P2P`: есть ли по нему API-документация

На момент анализа публичной официальной API-документации для `P2P Market` не найдено.

Что найдено официально:

- раздел справки `P2P` с пользовательскими инструкциями;
- правила, лимиты, комиссии;
- сценарии покупки, продажи, офферов, споров и ограничений.

Что не найдено официально:

- endpoint'ы для создания P2P-ордеров извне;
- endpoint'ы для чтения офферов;
- webhook'и для P2P;
- partner API для управления P2P из внешней платформы.

Итог:

- `P2P` в Crypto Bot выглядит как встроенный продукт для ручной работы пользователя внутри Telegram;
- для автоматизации внешней платформы опираться на `P2P` как на публичный API нельзя;
- для автоматизации платежей надо использовать `Crypto Pay`.

## Что такое `Crypto Pay`

`Crypto Pay` — это платежная подсистема на базе `Crypto Bot`, предназначенная для приема криптооплаты и выплат из внешних сервисов.

Официальная модель интеграции:

- внешний сервис создает счет;
- пользователь оплачивает его внутри инфраструктуры Crypto Bot;
- сервис получает подтверждение через webhook или опрос API;
- при необходимости сервис делает payout через `transfer`.

Это и есть основной официальный интеграционный слой для проектов, ботов и платформ.

## Базовая схема API

### Mainnet

`https://pay.crypt.bot/api/<method>`

### Testnet

`https://testnet-pay.crypt.bot/api/<method>`

### Авторизация

Токен передается в HTTP-заголовке:

`Crypto-Pay-API-Token: <TOKEN>`

Токен приложения создается в:

- `@CryptoBot` для mainnet;
- `@CryptoTestnetBot` для testnet.

Путь в интерфейсе:

`Crypto Pay -> Create App`

## Поддерживаемые способы работы

Официально поддерживаются:

- `GET`;
- `POST`;
- `application/json`;
- `application/x-www-form-urlencoded`;
- `multipart/form-data`.

Ответ API всегда приходит JSON-объектом с полем:

- `ok: true` при успехе;
- `ok: false` при ошибке.

При успехе полезная нагрузка находится в `result`, при ошибке в `error`.

## Методы `Crypto Pay API`

По официальной документации доступны следующие методы:

- `getMe`
- `createInvoice`
- `deleteInvoice`
- `createCheck`
- `deleteCheck`
- `transfer`
- `getInvoices`
- `getChecks`
- `getTransfers`
- `getBalance`
- `getExchangeRates`
- `getCurrencies`
- `getStats`

Ниже прикладной разбор каждого блока.

## 1. Служебные методы

### `getMe`

Нужен для проверки токена приложения и базовой информации о приложении.

Когда полезен:

- первичная проверка интеграции;
- health-check;
- smoke test после деплоя.

### `getCurrencies`

Возвращает список поддерживаемых валют.

Когда полезен:

- не хардкодить список активов;
- валидировать доступность валюты при создании счета;
- синхронизировать справочник валют в вашей платформе.

### `getExchangeRates`

Возвращает курсы обмена.

Когда полезен:

- если вы выставляете цену в фиате, но принимаете оплату в крипте;
- для внутреннего расчета лимитов;
- для UI-подсказок клиенту.

## 2. Инвойсы

### `createInvoice`

Главный метод для приема оплаты.

Что позволяет:

- создать счет в криптовалюте;
- создать счет в фиате;
- разрешить оплату одной или несколькими криптовалютами;
- ограничить время жизни счета;
- прикрепить служебный `payload`;
- добавить скрытое сообщение;
- настроить кнопку после оплаты;
- использовать `swap_to` для сценариев конвертации, если это нужно вашему кейсу.

Практический смысл:

- это основной метод для оплаты заявок;
- на каждую заявку можно создавать отдельный счет;
- `payload` удобно использовать как внешний идентификатор заявки.

### `getInvoices`

Нужен для чтения списка счетов и их статусов.

Можно фильтровать по:

- валюте;
- списку `invoice_id`;
- статусу;
- пагинации.

Практический смысл:

- резервный способ сверки, если webhook пропущен;
- nightly reconciliation;
- повторная синхронизация после сбоев.

### `deleteInvoice`

Удаляет созданный счет.

Практический смысл:

- закрывать устаревшие или ошибочные счета;
- запрещать дальнейшую оплату по старой заявке после замены реквизитов или суммы.

## 3. Checks

### `createCheck`

Создает чек на сумму в определенном активе.

Особенности:

- можно закрепить за `user_id`;
- можно закрепить за `username`.

Практический смысл:

- выдача бонусов;
- ручные компенсации;
- промо-механики;
- точечная отправка средств без инвойса.

### `getChecks`

Позволяет читать созданные чеки и их статусы.

### `deleteCheck`

Удаляет чек, если он больше не должен быть активен.

## 4. Выплаты

### `transfer`

Это метод для выплат пользователям из баланса вашего приложения.

Что важно:

- метод выключен по умолчанию;
- его нужно отдельно включить в `Security -> Transfers` внутри `Crypto Pay`;
- пользователь должен уже использовать `@CryptoBot`;
- нужен `Telegram user_id`;
- есть `spend_id` для идемпотентности.

Практический смысл:

- возвраты;
- выплаты партнерам;
- реферальные начисления;
- автоматическая выдача средств после согласования заявки.

Критически важный нюанс:

- если у вас нет надежного способа получить и хранить `Telegram user_id` клиента, payout-сценарий будет неполным.

### `getTransfers`

Позволяет читать историю выплат и сверять выполнение payout-сценариев.

Практический смысл:

- аудит;
- повторная проверка статуса после сетевых ошибок;
- reconciliation по `spend_id`.

## 5. Баланс и статистика

### `getBalance`

Возвращает баланс приложения.

Практический смысл:

- проверка достаточности средств перед payout;
- мониторинг остатка;
- алерты на низкий баланс.

В документации отдельно фигурируют:

- `available`;
- `onhold`.

Это важно для логики выплат: технически баланс может быть не нулевым, но часть средств может быть заблокирована.

### `getStats`

Возвращает статистику приложения.

Что особенно полезно:

- оборот;
- конверсия;
- число уникальных плательщиков;
- количество созданных и оплаченных инвойсов.

Практический смысл:

- BI-аналитика;
- контроль эффективности платежного сценария;
- проверка воронки "создана заявка -> выставлен инвойс -> счет оплачен".

## Webhooks

Официально у `Crypto Pay API` есть webhook-механизм.

На момент анализа в документации явно описан тип события:

- `invoice_paid`

Webhook приходит как HTTPS `POST` с JSON.

В payload передается объект `Invoice`.

### Ретраи и надежность

По состоянию на 18 марта 2026:

- повторная доставка webhook выполняется до 17 раз;
- окно доставки растянуто до 3 дней;
- если endpoint так и не отвечает, webhook'и автоматически выключаются;
- владелец приложения получает уведомление.

Это важное изменение: webhook стал надежнее, но теперь ваша система обязана мониторить не только успешную обработку, но и факт автоматического отключения webhook'ов на стороне Crypto Bot.

### Проверка подлинности webhook

Официально рекомендуется:

- использовать secret path в URL;
- проверять заголовок `crypto-pay-api-signature`;
- сверять HMAC-SHA-256 подпись тела запроса;
- секрет для HMAC строится из SHA-256 хэша токена приложения.

Практический вывод:

- нельзя слепо доверять входящему webhook;
- webhook handler должен проверять подпись до изменения статуса заявки;
- тело запроса нужно проверять в исходном виде, без неуправляемой промежуточной модификации.

## Комиссии и лимиты

### `Crypto Pay`

По официальной справке комиссия зависит от оборота приложения за последние 30 дней и стартует с `3%`, снижаясь по мере роста оборота.

Практический смысл:

- если вы строите бизнес-модель на оплате заявок, комиссию нужно включать в unit-экономику;
- если у вас низкомаржинальные заявки, это может быть существенным ограничением.

### `P2P Market`

По официальной справке:

- для покупателей комиссия `0%`;
- для продавцов `1%`;
- минимальный ордер `2 USD`;
- максимальный ордер `100,000 USD`.

Но это относится к пользовательскому P2P-сценарию внутри Crypto Bot, а не к внешнему открытому API.

## Ограничения и риски интеграции

### 1. Нет публичного API для `P2P`

Это ключевое ограничение.

Если ваша платформа заявок предполагает:

- автоматическое создание P2P-ордеров;
- чтение офферов;
- выбор лучшего оффера;
- автоматическое завершение P2P-сделок через backend;

то на официальном API этого сейчас не видно.

### 2. Custodial-модель

Расчеты идут внутри инфраструктуры Crypto Bot.

Это означает:

- вы зависите от правил платформы;
- вы зависите от их лимитов, AML/CFT и внутренних блокировок;
- это не прямой non-custodial blockchain flow;
- часть операций может быть ограничена политиками сервиса.

### 3. Выплаты зависят от Telegram user identity

Для `transfer` нужен `Telegram user_id`, и пользователь должен быть пользователем `@CryptoBot`.

Следствие:

- если клиент не стартовал бот, автоматический payout невозможен;
- в UX вашей платформы нужно предусмотреть шаг "привязать Telegram / запустить @CryptoBot".

### 4. Изменяемость API и правил

В политике `Crypto Pay` указано, что система может менять API и техническую документацию без предварительного уведомления партнеров.

Следствие:

- нужна регулярная проверка changelog;
- нужно закладывать адаптацию контрактов;
- нельзя считать интеграцию полностью стабильной "на годы без сопровождения".

### 5. KYC / AML / Holds

В пользовательской документации видно, что часть сценариев может упираться в AML-risk, holds и верификацию.

Следствие:

- для high-risk платежей или спорных операций нужно заранее продумать регламенты поддержки;
- нельзя обещать пользователю абсолютную мгновенность всех операций.

## Когда `@send / Crypto Pay` подходит для вашей платформы заявок

Подходит, если вам нужно:

- выставлять клиенту крипто-счет по заявке;
- получать подтверждение оплаты;
- автоматически переводить заявку в статус "оплачено";
- делать криптовыплаты пользователю;
- вести статусы платежей и сверку.

Не подходит как основной интеграционный слой, если вам нужно:

- программно управлять P2P market;
- строить внешний автотрейдинг поверх их биржи;
- полностью управлять кошельком пользователя;
- использовать это как универсальный API для всех внутренних модулей Crypto Bot.

## Рекомендуемая модель интеграции для платформы заявок

Если ваш проект связан с обработкой заявок, разумная архитектура выглядит так:

### Сценарий оплаты

1. В вашей платформе создается заявка.
2. Для заявки создается внутренний `payment_intent`.
3. Через `createInvoice` создается счет в `Crypto Pay`.
4. В вашу БД сохраняются:
   - `invoice_id`
   - `hash`
   - `payload`
   - валюта
   - сумма
   - срок жизни
   - ссылка на счет
5. Клиент оплачивает счет.
6. Ваш webhook принимает `invoice_paid`.
7. Система валидирует подпись.
8. Система идемпотентно переводит заявку в статус `paid`.
9. Если webhook пропущен, фоновый reconciliation добирает статус через `getInvoices`.

### Сценарий выплаты

1. В заявке появляется основание для выплаты.
2. Система проверяет, есть ли у клиента `telegram_user_id`.
3. Система проверяет баланс через `getBalance`.
4. Система вызывает `transfer` с уникальным `spend_id`.
5. Ответ сохраняется в таблицу выплат.
6. Повторная отправка с тем же `spend_id` не должна создавать дубль.
7. Фоновая сверка использует `getTransfers`.

## Архитектурные рекомендации

### 1. Делать интеграцию идемпотентной

Обязательно:

- уникальный индекс на `provider_invoice_id`;
- уникальный индекс на `spend_id`;
- обработка webhook через "already processed" guard.

### 2. Не делать webhook единственным источником истины

Нужно:

- webhook для realtime;
- `getInvoices` / `getTransfers` для reconciliation.

### 3. Развести бизнес-статусы и платежные статусы

Пример:

- статус заявки: `new`, `awaiting_payment`, `paid`, `processing`, `completed`, `cancelled`;
- статус платежа: `created`, `active`, `paid`, `expired`, `failed`.

### 4. Хранить сырой payload провайдера

Это нужно для:

- аудита;
- разборов спорных кейсов;
- восстановления логики после изменений документации.

### 5. Отдельно мониторить webhook-инфраструктуру

Нужны:

- алерт на рост необработанных webhook;
- алерт на mismatch подписи;
- алерт на долгое отсутствие событий при наличии активных счетов.

## Практический итог по `P2P`

Если вопрос именно про `P2P`, то ответ такой:

- официальные пользовательские документы есть;
- официального публичного API для внешней автоматизации не видно;
- использовать `P2P` как программный backend вашей платформы сейчас нельзя считать подтвержденным интеграционным сценарием.

Если вопрос про API интеграции в целом, то ответ такой:

- интегрироваться нужно через `Crypto Pay API`;
- для заявок это обычно достаточно, если задача сводится к оплате, сверке и выплатам;
- если вы рассчитывали управлять всем функционалом Crypto Bot программно, официальные материалы этого не подтверждают.
