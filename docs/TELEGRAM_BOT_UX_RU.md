# Telegram Bot UX

Дата: 20 мая 2026

## 1. Назначение

Telegram-бот — это пульт управления агентом и рабочий экран оператора.

Бот не должен хранить бизнес-логику захвата заявок. Он должен:

- показывать состояние агента;
- показывать активные заявки;
- запускать и ставить агента на паузу;
- менять лимит активных заявок;
- менять фильтр суммы `min/max`;
- закрывать конкретную заявку через `Оплачено`;
- давать быстрый доступ к ручным действиям.
- не давать оператору штатно пропускать уже взятую заявку.

## 3. Команды

Команд должно быть минимум. Основная работа идет через inline-интерфейс.

- `/start` — открыть панель агента;
- `/panel` — повторно открыть панель агента.

## 4. Главный экран

Показывает:

- статус агента;
- количество активных заявок;
- лимит активных заявок;
- свободные слоты;
- список активных заявок.

Кнопки:

- `Запустить` или `Пауза`;
- `Обновить`;
- `Активные`;
- `Лимит`;
- `Суммы`;
- `Сессия`;
- `Помощь`.

## 5. Экран активных заявок

Для каждой заявки есть действия:

- `Проверить оплату <id>`;
- затем отдельный экран `Подтвердить оплачено`.

Правило:

- `Оплачено` удаляет заявку только из активного списка после второго подтверждения;
- кнопки `Пропустить` нет, потому что взятая и неоплаченная заявка ведет к блокировке по времени;
- в production-версии запись останется в БД и истории.
- карточка с `QR` и ссылкой оплаты отправляется в бот только после `owned_confirmed` (подтверждено, что заявка действительно наша).
- события только из общей ленты сокета (`list:update add`) не считаются подтверждением владения и не должны запускать оплату.
- в карточке активной заявки должен быть таймер до автоотмены (`3:00` от подтвержденного захвата) и алерты `T-60`, `T-30`, `T-10`.

## 6. Экран лимита

Доступные быстрые значения:

- `1`
- `2`
- `3`
- `5`
- `10`

Правило:

- лимит применяется сразу;
- если активных заявок стало столько же, сколько лимит, агент переходит в состояние `capacity_reached`;
- после освобождения слота агент возвращается в `waiting`.

## 7. Экран фильтра суммы

Доступно:

- быстрые пресеты суммы;
- ручной ввод сообщением в формате `min max`;
- пример: `100 500`.

Правило:

- фильтр применяется до попытки `claim`;
- заявка вне диапазона не должна захватываться.

## 8. Обновление сессии через бот

Сценарий:

- оператор открывает DevTools в браузере;
- копирует curl или cookie string;
- отправляет сообщение боту;
- бот извлекает `access_token` и `__cf_bm`;
- бот показывает только наличие токенов, но не сами значения;
- бот пытается удалить сообщение с секретами.

Какой cURL копировать:

1. Открыть в браузере `https://app.send.tg/p2c/orders`.
2. Открыть DevTools через `F12`.
3. Перейти во вкладку `Network`.
4. Включить фильтр `WS` или найти запрос по строке `p2c-socket`.
5. Выбрать запрос с URL `wss://app.send.tg/internal/v1/p2c-socket/?EIO=4&transport=websocket`.
6. Нажать правой кнопкой по запросу.
7. Выбрать `Copy` -> `Copy as cURL (bash)`.
8. Отправить этот cURL боту в экран `🔐 Сессия`.

Важно:

- обычный curl страницы `/p2c/orders` может содержать только `__cf_bm`;
- socket curl должен содержать `access_token`;
- для WebSocket listener нужен `access_token`, а Cloudflare cookie нужна для прохождения защитного слоя.
- кнопка `Тест сокета` делает только короткую проверку подключения и не захватывает заявки.
- если в скопированном cURL нет `access_token=`, это не тот запрос для обновления socket-сессии.

## 8.1 PostgreSQL-хранилище сессии

Сессия платформы должна храниться не в памяти, а в PostgreSQL:

- таблица: `platform_sessions`;
- запись: одна актуальная сессия с `id = 1`;
- поля: `access_token_encrypted`, `cf_bm_encrypted`, `updated_at`;
- шифрование выполняется в приложении через `SESSION_ENCRYPTION_KEY`;
- в БД не должны попадать открытые значения `access_token` и `__cf_bm`.

Конфигурация:

```env
DATABASE_URL=postgresql://agent_user:change_me@localhost:5432/agent_db
SESSION_ENCRYPTION_KEY=replace_with_generated_key
```

Генерация ключа:

```bash
py -3 -m app.workers.generate_session_key
```

Для Docker Compose сервисы получают внутренний `DATABASE_URL` на host `postgres`, а локальный Windows-запуск использует `localhost`.

## 9. Доступ

Если `TELEGRAM_ADMIN_IDS` пустой:

- бот доступен всем, кто знает токен/бота.

Если `TELEGRAM_ADMIN_IDS` задан:

- бот отвечает только указанным Telegram user ids.

Пример:

```env
TELEGRAM_ADMIN_IDS=123456789,987654321
```

## 10. Проверка вручную

1. Подставить токен:

```env
TELEGRAM_BOT_TOKEN=...
TELEGRAM_ADMIN_IDS=...
```

2. Запустить:

```bash
py -3 -m app.bot.runner
```

3. В Telegram проверить:

- `/start`;
- кнопку `Запустить`;
- кнопку `Лимит`;
- кнопку `Суммы`;
- кнопку `Сессия`;
- изменение лимита;
- кнопку `Активные`;
- кнопку `Пауза`.

## 11. Что будет заменено позже

In-memory state будет заменен на:

- Redis active set;
- PostgreSQL history;
- реальные события из socket listener;
- реальные claim results.

UX при этом должен остаться тем же.
